Desert Modern Commerce

Project structure summary prepared for a presentation walkthrough. This document explains how the repository is organized, what each major folder contains, and how the frontend and backend connect at runtime.

Repository type: full-stack monorepo with npm workspaces
Main apps: client (React storefront + admin UI) and server (Express API)
Main objective: e-commerce platform with authentication, cart, checkout, orders, admin management, analytics, and deployment support

1. High-Level Architecture

Root Workspace Coordinates the monorepo, shared scripts, formatting, local development, and deployment entry points.
Client App React + TypeScript + Vite frontend for customers and administrators, with routing, state management, and API communication.
Server App Node.js + Express + MongoDB backend handling business logic, security, persistence, and third-party integrations.
Generated folders like node_modules/, client/dist/, and server/dist/ are build artifacts and are not part of the handwritten business logic.

3. What the Root Folder Does

File / Folder Role in the project
package.json Declares npm workspaces and global scripts such as npm run dev, build, lint, seed, and stripe:listen.
README.md Main project documentation: setup, scripts, environment variables, features, deployment, and Stripe local testing.
docs/ Supporting material for API testing and presentation, including the Postman collection.
scripts/stripe-listen.sh Helper script that forwards Stripe webhook events to the local Express endpoint during development.

4. Frontend Structure: client/

The frontend is a React + TypeScript application built with Vite. It contains both the customer storefront and the admin dashboard.

Path Purpose
client/src/main.tsx Frontend entry point. Wraps the app with the router, React Query provider, and toast notifications.
client/src/App.tsx Central route map. Defines public pages, protected user pages, and admin-only pages with lazy loading.
client/src/pages/ Page-level screens such as home, shop, product details, cart, checkout, login, register, profile, and admin pages.
client/src/components/ Reusable UI building blocks. It is split into common pieces, layout wrappers, and lower-level UI components.
client/src/store/ Zustand stores for authentication state, cart state, and UI state. Guest cart data is persisted in local storage.
client/src/hooks/ Custom hooks such as app bootstrapping, theme syncing, and debounce logic.
client/src/lib/ Shared client utilities, including the Axios API instance and React Query configuration.
client/src/types/ and validation/ Shared TypeScript definitions and form validation logic used by pages and components.

Frontend runtime flow

  1. main.tsx mounts the application and registers global providers.
  2. App.tsx declares routes and wraps them with error handling and suspense.
  3. useAppBootstrap.ts fetches the current user and cart when the app starts.
  4. lib/axios.ts sends authenticated API calls and automatically retries after token refresh.
  5. Pages render business features by composing reusable components and shared stores.

5. Backend Structure: server/

The backend is an Express API connected to MongoDB through Mongoose. It handles security, authentication, product management, cart operations, order creation, admin actions, and external services.

Path Purpose
server/src/server.ts Startup entry point. Connects to MongoDB and starts the HTTP server.
server/src/app.ts Express application definition. Registers security middleware, JSON parsing, health check, webhook route, API routes, and global error handling.
server/src/config/ Infrastructure configuration for environment variables, database, email, Stripe, and Cloudinary.
server/src/routes/ Route declarations that group endpoints by domain: authentication, products, categories, cart, orders, uploads, admin, and users.
server/src/controllers/ Business logic for each route. Controllers validate input, call models and utilities, and shape API responses.
server/src/models/ MongoDB data models for users, products, categories, carts, and orders.
server/src/middleware/ Reusable request middleware for authentication, uploads, and centralized error handling.
server/src/utils/ Shared backend helpers such as token generation, cookies, cart merging, email templates, and async error wrappers.
server/src/scripts/seed.ts Seeds the database with demo users and sample commerce data.
server/src/tests/api.integration.test.ts End-to-end style integration tests using an in-memory MongoDB instance.

Backend request flow

  1. The server starts in server.ts after a successful database connection.
  2. app.ts applies CORS, Helmet, rate limiting, cookies, body parsing, and sanitization.
  3. Requests are sent to domain routes such as /api/auth or /api/products.
  4. Controllers execute business logic and interact with Mongoose models.
  5. Shared utilities support token handling, emails, cart synchronization, and error formatting.

6. Functional Modules Inside the Project

Authentication Email verification Password reset Product catalog Cart sync Stripe checkout Order management Admin dashboard Analytics Cloudinary uploads

The structure of the repository reflects clear business domains. Each domain is represented on both sides of the application: page-level UI and stores on the frontend, then routes, controllers, models, and utilities on the backend.

7. Data and Integration Flow

  1. The user interacts with pages inside client/src/pages.
  2. The frontend calls the backend through the Axios client in client/src/lib/axios.ts.
  3. The backend exposes REST endpoints under /api/*.
  4. Business logic is executed inside controllers and persisted in MongoDB through Mongoose models.
  5. Stripe handles checkout, Cloudinary handles media uploads, and Nodemailer handles emails.
  6. Responses return to the frontend, which updates React Query caches and Zustand stores.

8. Suggested Presentation Walkthrough

For a professor presentation, the clearest order is to move from the outside of the repository toward the business logic:

  1. Start at the root folder and explain that this is an npm workspace monorepo.
  2. Show README.md and package.json to explain setup and scripts.
  3. Open client/ and present the frontend structure: entry point, routes, pages, components, stores.
  4. Open server/ and explain the backend layering: routes, controllers, models, middleware, utilities.
  5. Finish with docs/, test coverage, and deployment files to show project maturity.
If you want to narrate the structure in one sentence: this repository separates presentation logic in client/, business and data logic in server/, and shared project orchestration at the root.
